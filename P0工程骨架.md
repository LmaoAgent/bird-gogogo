# 《蒜鸟冲冲冲》P0 工程骨架与任务拆解

> **⚠️ 状态（2026-07-18）**：P0 核心闭环**已在 `prototype/`（纯 JS+Canvas）跑通**，美术已交付 `art_delivery/p0/`。本文 §5 的 T01–T13 拆解**大部分已完成，不要重新执行**；当前派工一律以 **`任务提示词.md`** 为准。本文现在的用途是 §1 工程结构与 §3 系统划分——迁 Cocos（任务提示词 T4）时照它分层。

> 面向程序，给出工程结构、系统划分与可分配任务。原则：**简单优先**——用「单例 Manager/System + 事件总线」，不引入 ECS/DI 重框架。数值全部来自配置表（见 `玩法数值与关卡设计.md`），代码只跑规则。

## 0. 技术栈
- **Cocos Creator 3.8+（LTS）+ TypeScript**；构建目标：微信小游戏。
- 竖屏 1080×1920，Canvas 适配 FIT_HEIGHT / SHOW_ALL。
- 状态管理：全局单例 + `EventBus`，UI 与逻辑事件解耦。

## 1. 工程目录结构
```
assets/
  scripts/
    core/                 # 框架层
      GameApp.ts             # 全局入口 + 主状态机
      EventBus.ts            # 事件总线
      ConfigLoader.ts        # JSON 配置表加载（预留远程热更）
      ObjectPool.ts          # 通用对象池（性能关键）
      Storage.ts             # 本地存档 + 微信云备份
      Analytics.ts           # 埋点封装（P0 先 console）
      AdManager.ts           # 广告封装（见 广告接入spec.md）
    game/                 # 玩法层
      systems/
        InputSystem.ts        # 拖动 → 队形中心 X
        CrowdSystem.ts        # 兵力 / 队形 / 渲染 / 池化（最核心）
        GateSystem.ts         # 门生成与结算
        CombatSystem.ts       # 对撞模型（玩法文档 §4）
        LevelManager.ts       # 关卡加载 / 波次 / 胜负 / 星级
      entities/
        GarlicBird.ts         # 单兵（骨骼动画封装）
        Gate.ts               # 门
        Enemy.ts              # 敌人 / BOSS
      GameController.ts       # 一局流程编排
    ui/                   # 界面层
      HomeUI.ts  HudUI.ts  ResultUI.ts  FailUI.ts
    defs/
      types.ts              # 门/关卡配置的 TS 类型（对齐玩法文档 schema）
      consts.ts             # 常量 / 事件名
  configs/                # 数据驱动配置表（JSON）
    levels/ level_01.json … level_10.json
    curve.json              # k(level)、H(level) 曲线
    gates.json  upgrade.json
  prefabs/                # 预制体
  art/                    # 美术资源（占位先行，命名见美术清单 §0）
```

## 2. 主流程状态机（GameApp）
```
Boot(加载配置+资源) → Home(主界面) → Game(一局) → Result/Fail(结算) → Home / 下一关
```
- 通过 `EventBus` 派发切换事件，场景/UI 订阅响应，逻辑不直接引用 UI。

## 3. 核心系统职责与接口（TS 签名草案）

| 系统 | 职责 | 关键接口 | 依赖 |
|---|---|---|---|
| `ConfigLoader` | 预加载全部 JSON 表，供各系统查询 | `load():Promise<void>` · `getLevel(n)` · `getCurve()` | — |
| `ObjectPool` | 单兵/门/特效全部池化（大军几百只，必须） | `get(key):Node` · `put(node)` | — |
| `InputSystem` | touch move → 归一化目标 X（世界坐标） | `onMove(cb)` · 参数 `dragSensitivity` | — |
| `CrowdSystem` | 维护逻辑兵力 `N`（可远大于渲染数）；队形 `R∝√N`；同屏上限 `N_render`，池化 + tint/scale 扰动；跟手平滑 | `addUnits(d)` · `setN(n)` · `getN()` · `update(dt)` | ObjectPool |
| `GateSystem` | 按配置在 `posZ` 生成门；触发线结算 add/mul/sub/div/pick；pick 按队形中心左右选 option；埋点 `gate_pick` | `spawn(cfg)` · `onTrigger(gate)` | ConfigLoader, CrowdSystem |
| `CombatSystem` | 玩法文档 §4：撞击比较 `N` 与 `H/k`，穿透/FAIL 结算 + 表现时长 `T`；BOSS 多波串联 | `resolve(wave):Result` | CrowdSystem |
| `LevelManager` | 加载关卡、推进波次（门序列+敌波）、胜负、星级（§6）；埋点 level_* | `loadLevel(n)` · `tick(dt)` | 上述全部 |
| `GameController` | 编排一局生命周期；复活流程（接 AdManager） | `startLevel(n)` · `revive()` | LevelManager, AdManager |
| `Storage` | 关卡进度/金币本地存档 + 微信云备份 | `save()` · `load()` | — |
| `Analytics` | 统一埋点，P0 console + 缓冲，第三方留 P1 | `track(evt, params)` | — |

> **CrowdSystem 是性能与手感的核心**：逻辑 `N` 与渲染数解耦，`N` 可达数千而只渲染 `N_render`（≈130）个实例，超出用缩放+数字表现；务必对象池化，禁止逐帧 new/destroy。

## 4. 数据流
```
configs/*.json → ConfigLoader → LevelManager 消费
   → 驱动 InputSystem / CrowdSystem / GateSystem / CombatSystem
   → CrowdSystem.N 状态 → HudUI 绑定渲染
```
全链路数值来自配置；改数值不改代码。
