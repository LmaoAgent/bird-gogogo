## V1 闸门突破体验修复:炸开瞬间不该秒人

```text
项目 /Users/mm/project/蒜鸟的战斗,原型在 prototype-v2/。真源:玩法数值与关卡设计v2.md §4.5。
跑法:python3 -m http.server 8097(或 .claude/launch.json 的 prototype-v2),?level=2 有两道闸门。
调试:window.__proto.step(帧数, g=>'left'|'center'|'right') 手动推进并绘制;__proto.goto(n) 跳关。

问题(老板实玩反馈):打闸门期间怪被拦在门后持续堆积(实测 120+ 只),打穿瞬间它们同时恢复前进,
按「每 3 只漏怪掉 1 兵」瞬间掉 40 兵 → 直接归零。**堆积越久死得越惨,等于惩罚了认真打门的玩家,方向是反的。**

任务:
1. **突破冲击波**(核心):闸门打穿瞬间,沿赛道向前释放一道冲击波,秒杀/重伤波及范围内的怪。
   强度与闸门 maxHp 挂钩(门越硬波越强),范围建议覆盖门后 15~20 个纵深单位。
   这既是保护也是爽点 —— 素材里打穿闸门就是一片炸开。
2. **分批释放**:冲击波没清掉的怪,按批次(建议每 0.15s 放行一批)恢复移动,不要同时涌出。
3. **突破后短暂强化**:打穿后给 1 秒左右的加速前进 + 无敌(或大幅降低掉兵),让玩家"冲出去"而不是"被埋住"。
4. 三个参数全部进 config/tuning.json(breakWaveRange/breakWaveDamage/releaseBatchInterval/breakBoostS),不写死。
5. 表现:冲击波要有明确的视觉(扩散环 + 强震屏 + 成片死亡特效),现在只有一个"突破!"飘字,不够。

红线:core/ 保持引擎无关(不引 DOM);伤害与胜负仍由 F / 单目标 DPS 决定,冲击波是规则的一部分要写进 core,
不是渲染层的障眼法;不改门/怪流的数值配置(那是 V7 的活)。
验收(主控核):① 关 2 两道闸门,打穿后掉兵不超过当时兵力的 25%(主控会用 __proto.step 实测前后 N);
② 冲击波扫倒一片有明确视觉反馈;③ 故意拖延打门(堆更多怪)不会导致必死 —— 堆积量与突破损失应弱相关;
④ 6 关体检仍全部通关、单局仍在 30-36 秒(跑 scratchpad 的 v2check 口径)。
```

## V2 小怪堆叠密度 + 基础物理碰撞

```text
项目 prototype-v2/。前置:V1 已合入。真源 §4.6。

问题(老板反馈):小怪要「密密麻麻堆叠在一起」,像参考素材里铺满赛道的红色海洋;
现在虽然按排生成了,但怪之间会互相穿模、堆积时不够挤,缺少物理感。

任务:
1. **怪之间的分离力**:每帧对邻近怪施加简单排斥(距离小于半径和时沿连线推开),
   让它们挤而不叠。为性能用网格分桶(spatial hash)做邻域查询,别 O(n²) —— 同屏目标 200+ 只。
2. **被拦时紧密堆叠**:闸门立着时,怪在门后应该一层层挤实(前排贴门、后排贴前排),
   形成素材里那种密不透风的墙,而不是稀疏停在同一条线上。
3. **撞进大军时的推挤**:怪接触大军时,大军单位被轻微推开再回位(纯表现即可),给"撞上了"的手感。
4. 渲染配合:按 z 排序绘制保证前后层次;远处怪允许轻微重叠以强化"海量"观感。
5. 新参数进 tuning.json(enemyRadius/separationForce/gridCell/maxEnemies 上调至 260+)。

红线:分离力只影响 x/z 位置表现与堆叠形态,**不得改变击杀结算与漏怪判定的口径**
(仍由 F 与 contactZ 决定,否则 V7 的配平全废);core 引擎无关;性能优先。
验收(主控核):① 闸门后堆积时铺满赛道、目视密不透风、无明显穿模;② 同屏 200 只时浏览器不掉帧
(主控会 step 到最密处看帧耗);③ v2check 六关结果与 V1 后基本一致(击杀/漏怪偏差 <15%,证明没动到判定口径)。
```

## V5 可破坏油桶(Last Z 机制)

```text
项目 prototype-v2/。前置:V4 已合入。参考:Last Z 的 barrels —— 打了给奖励,但要权衡值不值。

背景:当前玩家的火力只有一个用途(清怪/开门),没有"该不该分神"的取舍。油桶补这一层。

任务:
1. 新增油桶实体(config 里 barrels 数组):`{id, posZ, x, hp, reward}`,reward 支持
   `{dim:'N'|'L'|'R'|'D', op, value}` 或临时 buff(限时穿透/暴击,持续秒数进 tuning)。
2. **取舍来自成本**:打油桶期间火力被分走(建议按比例分流而非全占),清怪速率下降 → 漏怪风险上升。
   贪心去打 vs 稳住清怪,这个决策必须真实存在(主控会验:满火力打桶时漏怪明显上升)。
3. 桶要放在**有代价的位置**(怪潮中段而非空场),否则白送就没有取舍。
4. 表现:桶有血条/裂纹反馈,炸开有明确奖励飘字。
5. 埋点:barrel_hit / barrel_break(level, rewardDim)。

红线:奖励不得强到破坏 V7 的火力曲线(单个桶收益建议 ≤ 该段火力的 15%);core 引擎无关。
验收(主控核):① 打桶确实分流火力、漏怪上升(数据可见);② 奖励量级不破坏关卡校验;③ 桶的位置有取舍感(不是白送)。
```
